AWS AI Security, Compliance, and Governance for AIF-C01

IAM FOR AI SERVICES
AWS Identity and Access Management (IAM) controls access to Bedrock, SageMaker, and other AI services. Least-privilege principle: grant only permissions required for each role. Service roles allow Lambda, SageMaker, and Bedrock to access other AWS services on your behalf. Resource-based policies on S3 buckets restrict which services can access training data. IAM conditions can restrict access by region, IP address, or MFA status. AWS Organizations Service Control Policies (SCPs) can restrict AI service usage across accounts.

AMAZON BEDROCK SECURITY
Bedrock encrypts data in transit with TLS and at rest with AWS KMS. Customer-managed keys (CMK) allow you to control encryption key lifecycle. VPC endpoints for Bedrock prevent traffic from traversing the public internet. Model invocation logging captures all inputs and outputs to CloudWatch or S3 for audit. CloudTrail records all Bedrock API calls for compliance and forensics. Bedrock does not use customer data to train foundation models.

DATA PROTECTION FOR AI WORKLOADS
Training data stored in S3 should use server-side encryption (SSE-S3 or SSE-KMS). S3 bucket policies should deny public access. Amazon Macie detects PII and sensitive data in S3 training datasets. Data should be classified before use in AI training. Retention policies should define how long training data and model artefacts are stored.

MODEL GOVERNANCE
Model versioning tracks which model version was used for each inference. Amazon SageMaker Model Registry provides a central repository for approved models. Model approval workflows require sign-off before production deployment. Model monitoring with SageMaker Model Monitor detects data drift and model quality degradation. Model cards document intended use, limitations, and evaluation results.

COMPLIANCE AND REGULATORY CONSIDERATIONS
GDPR: AI systems processing EU personal data must comply with data subject rights including the right to explanation. HIPAA: PHI used in AI training must meet HIPAA security rule requirements; AWS provides a HIPAA-eligible environment. SOC 2: AWS services including Bedrock are SOC 2 compliant. AI governance frameworks should map controls to compliance requirements.

NETWORK SECURITY FOR AI
VPC isolation separates AI workloads from public networks. SageMaker can run training jobs and endpoints within a VPC. Private link enables private connectivity to Bedrock without public internet exposure. Security groups control inbound and outbound traffic to AI compute resources. Network ACLs provide an additional layer of subnet-level filtering.

THREAT MODELLING FOR AI SYSTEMS
Prompt injection: malicious inputs attempt to override model instructions. Mitigation: input validation, guardrails, and output filtering. Data poisoning: adversarial manipulation of training data. Mitigation: data provenance tracking and anomaly detection. Model theft: extracting model weights or recreating behaviour through repeated queries. Mitigation: rate limiting and usage monitoring. Sensitive data exposure: models memorising and repeating training data. Mitigation: differential privacy and output scanning.

AWS AI SECURITY SERVICES
Amazon GuardDuty: threat detection for AWS accounts including anomalous API calls to AI services. AWS Security Hub: centralised security posture across AI workloads. Amazon Inspector: vulnerability assessment for compute running AI inference. AWS Config: tracks configuration changes to AI resources for compliance. CloudWatch: monitors AI service usage, costs, and error rates. AWS Audit Manager: automates evidence collection for AI compliance audits.

GENERATIVE AI SPECIFIC SECURITY RISKS
Hallucination: model generates plausible but false information. Mitigation: retrieval-augmented generation (RAG) grounds responses in verified documents. Jailbreaking: users bypass safety filters through crafted prompts. Mitigation: Bedrock Guardrails and regular red-team testing. Indirect prompt injection: malicious instructions embedded in documents retrieved by RAG. Mitigation: input sanitisation before passing retrieved content to the model.

GOVERNANCE FRAMEWORKS
AI governance should define: ownership and accountability for AI systems, risk classification for AI use cases, approval processes for new AI deployments, ongoing monitoring and review cadence, incident response procedures for AI failures. AWS Well-Architected Framework Machine Learning Lens covers security pillar for ML workloads including threat modelling, access control, and audit logging.
